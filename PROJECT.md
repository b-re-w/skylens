# PROJECT.md — SkyLens 중간 평가 프로토타입

> 최종 평가에서는 실제 드론을 띄워 KOREN·Core HPC·Edge VM으로 이어지는 풀 파이프라인을 시연한다.
> 중간 평가(약 1개월)는 하드웨어·인프라 없이도 **핵심 경험(3D 복원 + AI 위험/사람 탐지 오버레이)이 "실제로 작동한다"는 확신**을 심사위원에게 주는 것이 목표다.
> 이 문서는 그 1개월짜리 프로토타입의 구현 계획을 상세히 서술한다.

---

## 0. 이 프로토타입이 증명해야 하는 것 / 증명하지 않아도 되는 것

중간 평가는 최종 평가의 축소판이 아니라 **다른 것을 증명하는 별도의 데모**다. 이 구분을 먼저 명확히 해야 구현 범위가 산다.

| | 증명 대상 | 이 프로토타입에서 |
|---|---|---|
| **최종 평가** | KOREN·Core HPC가 실제로 필요하다 (성능 대조) | ❌ 증명 안 함 |
| **최종 평가** | 실제 드론이 실시간으로 영상을 보낸다 | ❌ 증명 안 함 (사전 촬영으로 대체) |
| **중간 평가** | 3D 복원 결과가 실제 현장을 정확히 재현한다 | ✅ 사전 촬영·gsplat으로 증명 |
| **중간 평가** | "드론이 날아가며 탐색 → 그 자리에 3D 공간이 생겨남 → AI가 위험/사람을 찾아냄"이라는 **서사가 시각적으로 성립한다** | ✅ 이 문서의 핵심 목표 |
| **중간 평가** | 지휘관 관점의 상황판이 실제로 쓸모 있어 보인다 (UX) | ✅ 2뷰 연동으로 증명 |

즉 중간 평가에서 심사위원이 봐야 하는 것은 **"기술이 실현 가능하다"**는 신뢰이지, **"인프라가 KOREN이어야만 한다"**는 논증이 아니다. 후자는 최종 평가 몫이다. 이 구분 때문에 아래 구현 계획에서는 KOREN/일반망 대조, Core HPC/Edge 분리, 실시간 스트리밍 같은 요소를 의도적으로 제외하고 **로컬에서 완결되는 Three.js 2-뷰 시뮬레이션**에 집중한다.

## 1. 데모 컨셉 요약

- **화면 구성**: 모니터(또는 태블릿) 2대, 나란히 배치
  - **뷰어 1 (좌)**: "드론 시뮬레이션 스페이스" — 남색·저채도·와이어프레임 느낌의 로우파이 렌더링. 3대의 드론이 경로를 따라 이동하며 공간을 탐색하는 장면. 카메라는 드론을 전지적 작가 시점(3인칭 추적)으로 따라다님.
  - **뷰어 2 (우)**: "실제 3D 복원 상황판" — 실사 텍스처의 가우시안 스플랫. 뷰어 1의 카메라 경로를 살짝의 시간차(lag)를 두고 따라가며, 드론이 지나간 자리만큼 포인트클라우드가 점점 드러난다(reveal). 사람이 탐지되면 그 지점으로 자동 줌인하고 탐지 마커·라벨을 띄운다.
- **연출 의도**: 뷰어 1은 일부러 "가짜처럼", "시뮬레이션처럼" 저품질로 보여줘서, 뷰어 2가 같은 공간을 실사 3D로 복원해내는 순간의 대비(reveal)를 극적으로 만든다. 즉 뷰어 1의 로우파이함은 결함이 아니라 뷰어 2를 돋보이게 하는 연출 장치다.
- **핵심 트릭**: 두 뷰어는 **같은 가우시안 스플랫 데이터**를 쓴다. 차이는 머티리얼/렌더링 방식과 "얼마나 보여줄지"뿐이다. 새로운 3D 에셋을 두 개 만드는 게 아니라, 하나의 소스를 두 가지 방식으로 소비하는 구조로 설계해서 제작 리소스를 아낀다.

## 2. 사전 준비 (구현 이전, 오프라인 작업)

1. **촬영**: 시연할 실내·모형 공간을 정하고, 미리 정할 드론 경로(프리셋 경로, §4 참조)를 따라 **카메라 3대(또는 스마트폰 3대)로 전체 경로를 촬영**한다. 실제 드론 대신 사람이 들고 걷거나 짐벌 리그를 써도 무방 — 목적은 시점 3개짜리 영상 확보.
2. **가우시안 스플래팅 생성**: COLMAP/GLOMAP으로 포즈 추정 → gsplat(Nerfstudio)으로 학습 → `.ply`(또는 `.splat`) export. ARCHITECTURE.md의 실제 파이프라인(GLOMAP→gsplat→ICP)을 축소판으로 그대로 적용 — 즉 이 사전 작업 자체가 "우리 파이프라인이 실제로 동작함"의 증거가 된다.
3. **탐지 라벨링**: 최종 파이프라인의 UNet 추론은 이 단계에서는 스킵하고, 사람 인스턴스가 위치할 지점 2~3곳을 수동으로 3D 좌표로 지정해 마커 데이터(JSON)를 만든다. (UNet이 이미 준비돼 있다면 실제 추론 결과를 써도 되지만, 필수는 아님 — §7의 확장 옵션 참조)
4. **좌표계 정리**: 스플랫의 월드 좌표와 드론 경로 좌표(§4)를 같은 좌표계로 맞춘다. 이게 안 맞으면 "드론이 지나간 자리에 복원이 생긴다"는 연출이 어긋난다.

## 4. 드론 경로 & 뷰어 1 구현

### 4.1 경로 데이터 구조

경로는 3D 웨이포인트(위치 + 선택적으로 바라보는 방향)의 배열로 정의하고, 드론별로 별도 파일을 둔다.

```js
// paths/drone_01.json
{
  "waypoints": [
    { "t": 0,  "pos": [0, 3, 0],   "look": [1, 0, 0] },
    { "t": 5,  "pos": [10, 3, 2],  "look": [1, 0.2, 0] },
    { "t": 12, "pos": [10, 4, 12], "look": [0, 0, 1] }
    // ...
  ]
}
```

- `t`는 초 단위 타임코드. 재생 시 웨이포인트 사이를 **Catmull-Rom 스플라인**으로 보간해 자연스러운 곡선 비행을 만든다.
- 3개 드론은 서로 다른 구역을 담당하므로 경로 파일도 3개(`drone_01/02/03.json`)로 분리하고 동시에 재생한다.

### 4.2 프리셋 경로 + 키보드 이탈

- 기본 동작은 프리셋 경로를 따라 자동 재생.
- 방향키(↑↓←→, 필요시 Q/E로 고도)를 누르면 **현재 조작 중인 드론 1대**(탭 키 등으로 전환 가능)가 프리셋 경로에서 일시적으로 이탈해 수동 조작된다.
- 키 입력이 일정 시간(예: 1.5초) 없으면, 현재 위치에서 **가장 가까운 미래의 프리셋 웨이포인트로 부드럽게 복귀**(재접속 트윈)한다. 이렇게 하면 발표자가 "저희가 라이브로 조작해도 무너지지 않습니다"를 시연할 수 있어 프로토타입이 아니라 실제 조작 가능한 시스템처럼 보이는 효과가 있다.
- 구현: 매 프레임 "자동 모드 / 수동 모드" 상태를 체크하는 간단한 상태 머신 하나면 충분 (`AUTO → MANUAL → RETURNING → AUTO`).

### 4.3 뷰어 1 카메라(전지적 작가 시점)

- 카메라는 드론 중 하나(또는 발표 시나리오상 "주인공 드론")를 **3인칭 추적**한다: 드론 뒤 약간 위에서, 드론의 진행 방향을 바라보도록 `lerp` + `lookAt`으로 매 프레임 갱신.
- 급격한 방향 전환에도 카메라가 튀지 않도록 위치·회전 모두 낮은 감쇠 계수로 스무딩.
- 다중 드론이 화면에 다 보이도록, 발표 흐름에 맞춰 "추적 대상 드론"을 전환하는 기능도 넣어두면 좋음(예: 숫자키 1/2/3).

### 4.4 로우파이 "시뮬레이션 스페이스" 룩

목적: 뷰어 2와 대비되는 저정보량·기하학적 느낌.

- 배경: 깊은 남색(`#0a1128` 계열) 단색 또는 은은한 그리드/파티클 배경.
- 지형·구조물: 가우시안 스플랫을 그대로 쓰지 않고, **스플랫 포인트를 다운샘플링한 포인트클라우드**를 와이어프레임/도트 형태로 표현. `PointsMaterial`에 `sizeAttenuation`과 옅은 발광(에미시브) 느낌의 색(청록~남색 그라데이션)을 줘서 "스캐닝 중"처럼 보이게 한다.
- 드론: 심플한 로우폴리 모델(또는 발광 코어 + 프로펠러 파티클)로 표현하고, 이동 궤적을 옅은 트레일 라인으로 남겨 "탐색하고 있다"는 느낌을 강조.
- 이 화면은 정보량을 일부러 줄이는 것이므로 텍스처·라이팅에 리소스를 쓰지 않아도 된다 — 오히려 단순할수록 연출 목적에 부합.

## 5. 뷰어 2 구현 — 실사 3D 복원 + Progressive Reveal

### 5.1 기본 로딩

- 사전 준비한 가우시안 스플랫(`.ply`/`.splat`)을 `gaussian-splats-3d` 등으로 로드해 Three.js 씬에 배치. 이 시점에는 전체가 로드는 돼 있되 **아직 아무것도 보이지 않는 상태**(전부 마스킹)로 시작한다.

### 5.2 Progressive Reveal (드론 경로 → 복원 영역)

핵심 아이디어: "이미 완성된 스플랫"을 **드론이 그 위치를 지나간 적이 있는지**를 기준으로 서서히 드러낸다.

구현 방식(둘 중 하나, 혹은 병행):

1. **거리 기반 마스크(셰이더 접근, 권장)**
   - 프래그먼트/버텍스 셰이더에 "지금까지 드론이 지나온 위치들의 목록"을 텍스처(1D 위치 배열을 인코딩한 데이터 텍스처)로 넘긴다.
   - 각 스플랫 포인트에 대해, 셰이더에서 가장 가까운 드론 지나온-위치까지의 거리를 계산 → 반경 R 이내면 보이게(`alpha`), 아니면 숨김. 매 프레임 지나온 경로가 늘어나므로 드러나는 영역도 늘어난다.
   - CPU에서 매 프레임 "지나온 웨이포인트 배열"을 갱신해 유니폼/텍스처로 GPU에 올리는 정도의 비용이라 성능 부담이 적다.
2. **청크 기반 사전 분할(단순 구현, 대안)**
   - 스플랫을 사전에 공간적으로 N개 청크로 나눠두고, 드론 경로가 특정 청크의 바운딩 박스에 진입하면 해당 청크를 `visible = true`로 전환 + opacity 0→1 페이드인.
   - 셰이더를 직접 안 짜도 되는 대신 경계가 각지게 보일 수 있음. 프로토타입 일정이 촉박하면 이쪽으로 먼저 구현하고 여유가 되면 1번으로 고도화하는 순서를 권장.

- **시간차(lag) 연출**: 뷰어 2가 참조하는 "드론 지나온 경로"는 뷰어 1의 실시간 위치가 아니라, **N초 지연된 버퍼**를 사용한다(예: 3~5초 딜레이 링버퍼). 이렇게 하면 "드론이 먼저 훑고 지나가면 뒤이어 3D가 피어난다"는 서사가 명확히 보인다.
- **드러나는 방식의 질감**: 단순 On/Off보다는, 갓 나타난 스플랫 포인트는 잠깐 반짝이거나(emissive flash) 아래에서 위로 서서히 알파가 올라오는 짧은 트랜지션을 주면 "지금 막 복원되고 있다"는 인상이 강해진다.

### 5.3 사람 탐지 연출

- 사전 준비한 탐지 마커(§2-3) 중, 해당 지점 주변이 **reveal되어 실제로 보이기 시작한 시점**에 맞춰 트리거된다(즉 아직 안 보이는 영역에 있는 사람은 미리 뜨지 않음 — reveal 진행도와 연동).
- 트리거되면:
  1. 마커 아이콘(예: 붉은/녹색 핀 + pulsing ring)과 "생존자 추정 위치" 라벨이 3D 공간에 나타남.
  2. 뷰어 2 카메라가 **자동으로 해당 좌표로 부드럽게 이동 + 줌인**(§8에서 다루는 카메라 상태 전환의 트리거).
  3. UI 오버레이(HTML/CSS, Three.js 캔버스 위에 얹는 패널)로 "위험구역 감지" 혹은 "사람 감지 — confidence 0.xx" 텍스트, 그리고 **"탐지 확인" 버튼**을 표시.
- **"탐지 확인" 버튼**을 누르기 전까지 카메라는 그 지점에 고정(§8 참조).

## 6. 위험구역 오버레이 (선택 — 시간 여유 시)

- ARCHITECTURE.md의 세그멘테이션 헤드(위험구역=stuff)에 대응하는 연출로, 특정 영역을 반투명 붉은 볼륨/디컬(decal)로 오버레이해 "이 구역은 위험"임을 표시.
- 우선순위는 사람 탐지보다 낮음 — 중간 평가 핵심 서사는 "탐색→복원→사람 탐지"이므로, 시간이 부족하면 이 항목은 발표 멘트("최종 평가에서는 여기에 위험구역 마스크도 함께 표시됩니다")로 대체 가능.

## 7. AI 파이프라인과의 연결 (진짜 vs 연출)

이 프로토타입에서 "AI가 탐지했다"는 것은 대부분 **사전에 좌표를 지정해둔 스크립트 트리거**다. 이는 속임수가 아니라 데모 설계의 정상적인 선택이며, 발표 시 명확히 구분해서 전달하는 편이 신뢰를 준다:

- **실제로 검증된 부분**: 가우시안 스플래팅 3D 복원 파이프라인(GLOMAP→gsplat) 자체는 §2에서 실제로 돌린 결과다. "3D 복원이 된다"는 절대 연출이 아니다.
- **연출된 부분**: 사람 탐지 마커의 좌표는 이번 프로토타입에서는 수동 지정. UNet 추론이 이미 어느 정도 진행됐다면, 실제 추론 결과(바운딩박스/인스턴스 좌표)를 그대로 마커 좌표로 흘려 넣는 것으로 대체 가능하며, 그 경우 발표에서 "AI가 실제로 탐지한 결과"라고 정확히 말할 수 있다.
- 발표 스크립트에는 이 경계를 숨기지 말고, "3D 복원은 실촬영 데이터로 검증했고, 탐지 AI는 최종 평가까지 정확도를 고도화할 예정"이라는 식으로 명확히 로드맵화하는 것을 권장.

## 8. 뷰어 1 ↔ 뷰어 2 카메라 동기화

### 8.1 요구사항 정리

- 평상시: 뷰어 2의 카메라는 뷰어 1의 카메라 상태(위치/회전/시간차 버퍼)를 따라간다.
- 사람 탐지 발생: 뷰어 2 카메라는 뷰어 1로부터의 동기화를 **일시 중단**하고, 탐지 지점으로 확대·고정.
- "탐지 확인" 버튼 클릭: 고정 해제 → 뷰어 1의 (그 시점 기준) 카메라 상태로 **스무스하게 복귀**(트윈) → 이후 다시 실시간 동기화 재개.

### 8.2 통신 방식에 대한 제안 — WebRTC 대신 BroadcastChannel

원 아이디어는 WebRTC로 카메라 state를 공유하는 것이었는데, 실제 구현 관점에서 몇 가지를 짚고 넘어가는 게 좋을 것 같다.

- WebRTC는 본래 **미디어 스트림(영상/오디오)이나 임의 P2P 데이터 채널**을 위한 프로토콜로, 시그널링 서버(STUN/TURN 포함) 설정이 필요하다. 여기서 옮기려는 데이터는 카메라 position/quaternion 같은 몇십 바이트짜리 JSON뿐이라, WebRTC의 시그널링 비용 대비 얻는 이득이 거의 없다.
- 두 뷰어가 **같은 브라우저에서 열리는 두 창/탭**이라면(모니터 2대에 각각 창을 띄우는 구성), 브라우저 표준 **`BroadcastChannel` API**로 충분하다. 동일 오리진의 여러 컨텍스트 간 메시지를 지연 없이 주고받을 수 있고, 서버·시그널링이 전혀 필요 없다.
- 두 뷰어를 **아예 하나의 앱**(한 페이지 안에 두 개의 `<canvas>`/Three.js renderer)으로 만드는 방법도 고려할 만하다. 이 경우 상태 공유는 통신이 아니라 그냥 같은 JS 객체를 참조하는 것으로 끝나 가장 단순하고 지연도 없다. 모니터 2대 구성은 OS 차원에서 창을 각 모니터로 드래그하거나 `window.open`으로 두 번째 창을 별도 모니터에 띄우는 방식으로 처리 가능.
- 만약 두 화면이 **물리적으로 다른 기기**(예: 랩탑 + 태블릿)에서 열려야 한다면 BroadcastChannel은 쓸 수 없고, 그때는 로컬 네트워크상의 경량 WebSocket 서버(예: 작은 Node 서버로 상태 브로드캐스트)가 WebRTC보다 구현이 훨씬 간단하고 안정적이다.

**권장안**: 모니터 2대가 같은 컴퓨터에 물려 있다면 **단일 앱 + 두 캔버스** 또는 **BroadcastChannel**로 시작하고, 태블릿처럼 별도 기기가 필요해지는 시점에 WebSocket으로 전환. 이 문서에서는 이후 설명을 "상태 공유 채널"이라는 추상으로 통일해서 서술한다(구현체는 위 셋 중 상황에 맞게 선택).

### 8.3 상태 머신

```
[SYNCED]  뷰어2 카메라 = f(뷰어1 카메라 상태, lag buffer)
   │  사람 탐지 트리거
   ▼
[FOCUSING]  뷰어2 카메라를 탐지 좌표로 트윈 이동/줌인 (뷰어1 동기화 무시)
   │  트윈 완료
   ▼
[LOCKED]  탐지 좌표에 카메라 고정, "탐지 확인" 버튼 표시
   │  버튼 클릭
   ▼
[RETURNING]  뷰어2 카메라를 "현재 시점의 뷰어1 상태"로 트윈 이동
   │  트윈 완료
   ▼
[SYNCED]  (복귀)
```

- `FOCUSING`/`RETURNING` 트윈은 급격한 컷 없이 ease-in-out으로, 시간은 1.5~2초 정도가 "지휘관이 상황을 인지할 시간"을 주면서도 발표 흐름을 안 늘어지게 하는 균형점으로 적당.
- `LOCKED` 상태에서도 뷰어 1은 계속 진행되므로(드론은 계속 날아감), `RETURNING` 시점의 목표는 **버튼을 누른 시점의 뷰어 1 최신 상태**여야 한다 — 탐지 발생 시점 상태로 되돌아가면 안 됨.
- 공유 채널로 넘기는 데이터는 최소한으로: `{ position: [x,y,z], quaternion: [x,y,z,w], timestamp }` 정도. 시간차(lag) 버퍼는 이 스트림을 뷰어 2 쪽에서 큐잉해 N초 전 상태를 꺼내 쓰는 방식으로 구현.

## 11. 최종 평가와의 연결고리 (참고)

이 프로토타입에서 만든 자산은 최종 평가에서 대부분 재사용 가능하다:

- 뷰어 1/뷰어 2 구조는 그대로 유지하되, 뷰어 1의 "프리셋 경로 재생"이 **실제 드론의 실시간 텔레메트리 수신**으로, 뷰어 2의 "사전 계산된 스플랫"이 **Core HPC에서 실시간으로 push되는 스플랫 청크**로 교체된다.
- 상태 공유 채널(§8.2)은 로컬 네트워크의 실제 WebSocket/이벤트 채널로 자연스럽게 확장.
- 사람 탐지 트리거는 수동 좌표 대신 실제 UNet 추론 + Depth Map 레이캐스팅 결과(ARCHITECTURE.md §3-A)로 교체.
- 즉 이 프로토타입은 "일회성 데모"가 아니라 **최종 파이프라인의 프론트엔드/뷰어 레이어를 미리 완성해두는 작업**이라는 점을 심사위원에게 강조할 수 있다.
